# Harness v2: AI-Generated Architectural Constraints

## Problem

Ship's current enforcement layer (Policy v1) uses hardcoded templates, grep-based pattern matching, and manually authored shell scripts. It provides safety guardrails but cannot enforce project-specific coding conventions. The rules are static, written by humans, and identical across projects.

AI agents are capable but structurally unreliable. 45% of AI-generated code has security flaws. The solution is not better prompting — it is mechanical enforcement. But the enforcement rules themselves should come from AI understanding of the project, not from templates.

## Solution

Replace the Policy v1 system with a unified Harness that uses AI to analyze each project's codebase, discover its conventions, and generate enforceable rules. Ship provides the infrastructure; AI provides the intelligence.

**Core principle: AI writes rules to constrain AI.** Ship provides no templates, no preset rules. Every rule is generated by AI from the project's actual code.

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Rule types | Structural (deterministic) + Semantic (LLM-judged) | Grep-able patterns need 100% reliability; coding conventions need semantic understanding |
| Structural enforcement | `command` hook → deny | Deterministic, fast, zero cost, hard block |
| Semantic enforcement | `agent` hook → additionalContext | Agent can Read rule files, understand context; soft feedback avoids false-positive blocks |
| Semantic model | claude-haiku-4-5-20251001 | Cost-effective for convention checking |
| Rule source | AI-generated from project code | No templates — every project is different |
| Rule storage | `.ship/rules/` (git tracked) | Reviewable, versionable, team-shared |
| Hook registration | `.claude/settings.json` (project-level) | Native Claude Code mechanism, git tracked |
| Activation | `/ship:harness` and `/ship:unharness` skills | User controls when enforcement is active |
| Scope | Project-level only | No org-level inheritance — each repo is independent |
| Failure behavior | Structural → deny, Semantic → feedback | Deterministic rules are safe to hard-block; probabilistic rules give guidance only |
| Policy v1 | Removed | Harness v2 subsumes all Policy v1 functionality |

## Architecture

```
┌─────────────────────────────────────────────────┐
│              .claude/settings.json              │
│  (project-level, git tracked)                   │
│                                                 │
│  PreToolUse hooks (exactly 2 entries):          │
│    ┌──────────────┐    ┌───────────────────┐    │
│    │ command hook  │    │   agent hook      │    │
│    │ enforce-      │    │   reads rules     │    │
│    │ structural.sh │    │   from .ship/     │    │
│    │               │    │   rules/semantic/ │    │
│    │ deny on fail  │    │   feedback only   │    │
│    └──────┬───────┘    └────────┬──────────┘    │
│           │                     │               │
└───────────┼─────────────────────┼───────────────┘
            │                     │
            ▼                     ▼
┌──────────────────┐  ┌────────────────────────┐
│ .ship/rules/     │  │ .ship/rules/           │
│ structural/      │  │ semantic/              │
│                  │  │                        │
│ *.sh scripts     │  │ *.md rule docs         │
│ (AI-generated)   │  │ (AI-generated)         │
└──────────────────┘  └────────────────────────┘

Deterministic · Fast · Deny  Probabilistic · Slower · Feedback

### File Structure

```

├── rules/

│   ├── enforce-structural.sh         # AI-generated router script
│   ├── structural/
│   │   └── *.sh                      # AI-generated check scripts
│   └── semantic/
│       └── *.md                      # AI-generated convention docs
├── audit/                            # Optional (AI decides if needed)
└── tasks/                            # Existing, unchanged

.claude/
└── settings.json                     # Hook entries (when harnessed)

### Hook Registration

Exactly two hook entries in `.claude/settings.json`, regardless of how many rules exist:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [{
          "type": "command",
          "command": "bash .ship/rules/enforce-structural.sh",
          "statusMessage": "Checking structural rules..."
        }]
      },
      {
        "matcher": "Write|Edit",
        "hooks": [{
          "type": "agent",
          "prompt": "You are a code convention enforcer. Read .ship/rules/rules.json to find all enabled semantic rules. For each applicable rule (check scope against the file being written), read the rule's .md file from .ship/rules/semantic/. Then verify the code in $ARGUMENTS follows those conventions. If violations found, return JSON with hookSpecificOutput.additionalContext describing each violation and how to fix it. If no violations, return nothing.",
          "model": "claude-haiku-4-5-20251001",
          "statusMessage": "Reviewing coding conventions..."
        }]
      }
    ]
  }
}
```

Adding/removing rules only changes files under `.ship/rules/` — hooks layer is stable.

### Rule Index (`rules.json`)

AI-generated. No fixed schema beyond:

```json
{
  "version": 1,
  "structural": [
    {
      "id": "<ai-chosen-id>",
      "name": "<ai-chosen-name>",
      "description": "<what this rule enforces>",
      "script": "structural/<script-name>.sh",
      "scope": "<glob pattern for applicable files>",
      "enabled": true
    }
  ],
  "semantic": [
    {
      "id": "<ai-chosen-id>",
      "name": "<ai-chosen-name>",
      "description": "<what this rule enforces>",
      "file": "semantic/<rule-name>.md",
      "scope": "<glob pattern for applicable files>",
      "enabled": true
    }
  ]
}
```

### Structural Rule Scripts

Each `.sh` script in `structural/` is AI-generated. It receives hook input JSON on stdin, checks one specific structural constraint, and exits non-zero with an error message if violated.

The router script (`enforce-structural.sh`) is also AI-generated. There is no Ship-provided template for this script. AI writes it based on the project's language, build tools, and directory structure. Its responsibility: read `rules.json`, match file paths against scopes, run applicable rule scripts, and emit deny JSON on failure.

### Semantic Rule Files

Each `.md` file in `semantic/` is AI-generated. It contains:
- Rule description (natural language)
- Correct code examples
- Incorrect code examples
- Scope information
- Rationale

No enforced format. The agent hook reads these files and uses its judgment to verify code compliance.

## Setup Flow

Three-step rule discovery during `/ship:setup`:

### Step 1: Infer from Code

AI analyzes the project codebase:
- Scans directory structure, identifies layering patterns (if any)
- Analyzes import/require graphs, discovers dependency boundaries
- Samples error handling, validation, logging patterns across files
- Identifies naming conventions (variables, files, functions)
- Detects security-sensitive patterns (credential access, env file usage)
- Assesses confidence: intentional convention vs. coincidence

No templates. AI decides what patterns exist based on evidence.

### Step 2: Supplement from Documentation

AI reads existing project documentation:
- CONTRIBUTING.md, STYLE_GUIDE.md, ARCHITECTURE.md
- Linter configs (.eslintrc, ruff.toml, tsconfig.json) for implicit conventions
- CLAUDE.md / AGENTS.md for coded behavioral rules

Extracts rules that Step 1 didn't discover from code patterns alone.

### Step 3: User Confirmation

Present discovered rules to the user for confirmation:
- Each rule shown with evidence (code references, frequency)
- User toggles rules on/off
- User adds rules AI missed (free-text description)
- User confirms with "done"

After confirmation, AI generates:
1. `rules.json` — rule index
2. `structural/*.sh` — check scripts for structural rules
3. `semantic/*.md` — convention docs for semantic rules
4. `enforce-structural.sh` — router script
5. Hook entries in `.claude/settings.json`

## Activation Skills

### `/ship:harness` — Enable

1. Check `.ship/rules/rules.json` exists (if not, prompt to run `/ship:setup`)
2. Read `.claude/settings.json`
3. Merge the two hook entries (preserve existing hooks)
4. Confirm: "Harness activated. N structural + M semantic rules enabled."

### `/ship:unharness` — Disable

1. Read `.claude/settings.json`
2. Remove the two harness hook entries (identify by command path / statusMessage)
3. Preserve `.ship/rules/` files
4. Confirm: "Harness deactivated. Rules preserved in .ship/rules/."

## Migration from Policy v1

The following components are replaced and can be removed:

| Removed | Replaced By |
|---------|-------------|
| `.ship/ship.policy.json` | `.ship/rules/rules.json` + rule files |
| `.ship/ship.policy.base.json` | Nothing (no org-level inheritance) |
| `bin/policy-boundaries.sh` | AI-generated structural rule scripts |
| `bin/policy-secrets.sh` | AI-generated structural rule for secret detection |
| `bin/policy-operations.sh` | AI-generated structural rule for dangerous commands |
| `bin/policy-context.sh` | Not needed (no policy to inject) |
| `bin/lib/policy.sh` | Not needed (no shared policy library) |
| `merge_policies` logic | Not needed (no org inheritance) |

Retained components:
- `bin/stop-gate.sh` — phase enforcement (orthogonal to rules)

## What Ship Provides vs What AI Generates

| Ship provides (infrastructure) | AI generates (intelligence) |
|---|----|
| Hook registration mechanism | Which rules to enforce |
| `settings.json` management | Rule content and detection logic |
| harness/unharness skills | Structural check scripts |
| Setup skill with analysis prompt | Semantic rule documentation |
| Execution framework (2 hook entries) | `enforce-structural.sh` router |
| `rules.json` schema (minimal) | `rules.json` content |

## Open Questions

1. **Rule evolution** — When the codebase evolves, how do rules stay current? Periodic re-scan? Manual `/ship:harness-update` skill?
2. **Agent hook cost** — Every Write/Edit triggers an agent (Haiku). For projects with high edit frequency, is this acceptable? Should there be a debounce or batch mechanism?
3. **Conflict with existing linters** — If the project already runs ESLint via pre-commit, should semantic rules avoid overlapping with what ESLint already checks?
